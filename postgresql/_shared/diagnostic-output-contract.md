# 诊断类技能统一输出契约

本文件定义所有诊断/判断类技能的最终报告必须遵循的结构。

**适用范围**：pg-runtime-risk、pg-bloat-root-cause、pg-load-spike-forensics、
pg-security-audit、pg-design-audit、pg-sql-audit、pg-host-resource-risk、
find-postgres-bug、pg-parameter-tuning-advisor、pg-sql-tuning-advisor 
等一切以"给出诊断结论/判断"为核心产出的技能。创作型、生成型技能不适用本契约。

**强度**：统一强度，不分日常巡检/重大排查场景，以下十个章节均为**必选**，
技能可在此基础上增补专属章节（例如某技能特有的评分表），但不可删减或简化
任何必选章节。

---

## 报告模板正文

```markdown
## 结论 / 观点
[一句话：症状是什么、根因判断是什么]

## 逻辑推演过程
[从现象到结论的推导链条，是"证据如何一步步导出结论"的推理路径，
不是证据罗列，也不是结论重述。
例：checkpoint 频繁 → 触发大量 full page write → IO 争抢 →
与卡顿时间窗口吻合 → 排除锁等待/网络因素后，收敛到 checkpoint 参数配置不当]

## 证据链
1. [具体数据/查询结果] → 支持了什么判断
2. [具体数据/查询结果] → 支持了什么判断
...
（每条证据须可回溯到具体 SQL/日志/快照来源，不接受"经验判断"类无源证据）

## 证据权威性
| 证据 | 来源类型 | 权威性 |
|---|---|---|
| 证据1 | 实时快照直接查询 | 高（一手数据） |
| 证据2 | 用户口述症状描述 | 中（未验证的间接信息） |
| 证据3 | 历史经验类比 | 低（非本次实测，仅供参考） |
（区分"一手实测数据" vs "推断/类比" vs "未经验证的用户描述"，
避免把强弱不同的证据混为一谈）

## 结论适合的边界
[这个结论在什么范围内成立：适用的 PG 版本、部署架构（单机/HA/云托管）、
负载特征等。明确说明"本结论不适用于"的场景，防止误用到不匹配的环境]

## 结论成立的前提条件（第一性原理拆解）
[把结论拆解到最基础的假设层，列出缺一不可的前提。
例：
- 前提1：checkpoint_timeout 与 max_wal_size 当前配置为默认值（未被自定义调整）
- 前提2：磁盘 IO 是本次瓶颈的唯一变量（假设网络、CPU 均正常）
- 前提3：卡顿期间无并发大事务/批量写入]

## 前提发生变化时的新结论
[针对上面每条前提，给出"如果前提不成立，结论会如何变化"，
形成条件分支而非单一死结论。
例：
- 若前提2不成立（发现同期有批量导入任务）→ 根因应指向应用层批处理调度，
  而非参数配置，建议动作随之改为"调整批处理时间窗口"而非"调整 checkpoint 参数"
- 若前提3不成立（发现长事务持锁）→ 应转向 pg-find-bloat / 锁等待分析，
  当前结论仅为并发因素之一，非唯一根因]

## 置信度
高/中/低，理由：[基于证据权威性和前提验证完整度综合判断——一手实测证据越多、
前提验证越充分，置信度越高；仅靠间接推断或前提未逐一验证，置信度应主动降级]

## 建议动作
- [ ] 立即执行：xxx（低风险，前提条件已充分验证）
- [ ] 需要你确认后执行：xxx（有数据丢失/锁表风险，或前提条件尚未完全排除其他可能）
- [ ] 仅供参考：xxx（置信度中低，或依赖未验证前提）

## 未覆盖 / 需人工判断的点
[技能力所不及、需要你亲自决策的部分——尤其是那些无法用只读查询验证的前提假设，
或需要业务上下文才能判断的边界情况]
```

## 生成报告前自检（技能自查用，不作为报告章节输出）

- [ ] 逻辑推演过程是否展示了从证据到结论的推导步骤，而非直接罗列证据？
- [ ] 每条证据是否标注了来源类型（一手实测 / 间接推断 / 用户描述）？
- [ ] 前提条件是否拆到了"缺一不可"的最小假设层，而非泛泛而谈？
- [ ] 每条前提是否都对应了"若不成立"的新结论？
- [ ] 置信度是否和证据权威性、前提验证完整度挂钩，而非孤立打分？

## 各技能接入方式

在各技能的 SKILL.md 正文末尾加入引用说明，例如：

```markdown
## 输出规范

本技能的最终报告须遵循 _shared/diagnostic-output-contract.md 定义的输出结构，
可根据本技能特点增补章节，但不可删减或简化"结论/逻辑推演/证据链/证据权威性/
边界/前提条件/前提变化后新结论/置信度/建议动作/未覆盖点"这些必选部分。
```

建议将本文件放置于共享目录（如 `_shared/diagnostic-output-contract.md`），
各技能通过引用而非复制，确保契约迭代时全部技能自动同步。
