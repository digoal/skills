# 报告模板：AWS Performance Insights 风格 PostgreSQL 性能诊断报告

生成报告时按以下结构与顺序输出，所有数值来自 Step 5/6 的查询结果，禁止编造。

---

## 📊 性能总览仪表盘

```
┌─────────────────────────────────────────────────────────────┐
│  分析时间段：<用户指定起止时间>                                │
│  实际快照窗口：Snap#<snap_begin_id> (<begin_time>) → Snap#<snap_end_id> (<end_time>) │
│  窗口长度：<window_seconds 换算为 时/分>                        │
│  平均活跃会话 (AAS)：<aas>                                     │
│  vCPU 数量：<vcpu，若未知则写"未提供，建议用户补充"> │
│  CPU 利用率 (AAS/vCPU)：<百分比，若vCPU未知则省略此行>           │
│  最大活跃会话 (峰值)：<peak_active_sessions> (出现在 <peak_time>) │
│  缓存命中率：<cache_hit_ratio_pct>%                            │
└─────────────────────────────────────────────────────────────┘
```

若 Step 2/4 触发了提示（reset 不一致已终止 / 窗口过短 / 窗口过长），在此仪表盘上方先输出提示语。

## 🔝 按维度负载 TOP SQL

依次输出四张表（总耗时、调用频率、IO 消耗、WAL 生成量；若 avg_latency 与 rows 维度也有意义可一并给出），每张表列：

| # | Query ID | SQL 摘要（≤200字符） | 指标值 | 占总量% | 调用次数 | 平均延迟(ms) |
|---|---|---|---|---|---|---|

## ⏳ 等待事件分布

用文字条形图 + 百分比呈现（CPU/IO/Lock/WAL/其他，合计≈100%）：

```
CPU 等待：       <条形> <pct>%
IO 等待：        <条形> <pct>%
锁等待：         <条形> <pct>%
WAL 写入等待：   <条形> <pct>%
其他：           <条形> <pct>%
```

若 `track_io_timing` 未开启或 PG 版本不支持某指标，在此明确标注"当前版本/配置不支持，跳过"。

| 数据库 | AAS 贡献 | 总耗时占比 | 事务数 | IO 读占比 |
|---|---|---|---|---|

## 👤 按用户/应用负载拆解

| 用户 | 应用 | AAS 贡献/总耗时占比 | 平均活跃连接数 | 峰值连接数 |
|---|---|---|---|---|

## 🎯 关键发现与根因分析

每条发现固定包含五个字段，缺一不可：

- **发现**：一句话描述现象
- **影响**：对系统性能的具体影响
- **根因**：基于 SQL 文本/执行特征推断的根因（引用 `root_cause_patterns.md` 中的判断逻辑）
- **证据**：引用具体的差值数据支撑（不得凭空断言）
- **建议**：具体可执行的优化操作（含 SQL 语句，如适用）
- **预估收益**：高/中/低 + 预估节省的数据库时间百分比（基于该 SQL 占总耗时的比例估算，需说明这是估算而非实测保证）

## 📈 负载趋势与异常检测

- 若窗口内有 3 个以上快照，用文字曲线呈现活跃会话数随时间变化，标注峰值时间点
- 列出 Step 6 中检测到的负载突增、延迟翻倍的 queryid、窗口内新出现的 queryid
- 每条异常都需说明"由谁/什么原因导致"（若能从数据中推断）

## ⚙️ 优化优先级排序 Top 5

| 优先级 | 优化项 | 预期收益 | 实施难度 | 建议执行窗口 |
|---|---|---|---|---|

按预估收益从高到低排序，P0 为最值得优先做的一项；每项标注实施难度（低=ONLINE操作，中=需改代码，高=需停机/架构调整）。

## 🧭 后续监控建议

- 建议持续关注的指标及阈值
- 建议的下一次深度分析时间窗口
- 若快照采集基础设施本身有不足（如采集频率过低、未采集 pg_stat_activity 等），给出改进建议

---

**通用要求**：全文中文输出；PG 版本/配置不支持的指标一律标注"跳过"而非编造；密码/连接串等敏感信息不得出现在报告中。
