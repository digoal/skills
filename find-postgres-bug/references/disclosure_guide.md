# 负责任披露指南

## 第一步：先确认没有被重复报告

在生成正式报告前，务必检索：

1. [PostgreSQL 邮件列表归档](https://www.postgresql.org/list/) — 搜索关键现象/函数名
2. [Commitfest](https://commitfest.postgresql.org/) — 看是否已有相关补丁在审阅中
3. [已知问题/发行说明](https://www.postgresql.org/docs/current/release.html) — 确认不是已知的、文档化的行为变更
4. Git 提交历史（`git log --all --grep=<关键词>`）— 确认目标分支上没有已经修复但你的源码目录尚未同步的提交

## 第二步：判断披露渠道

| 情形 | 渠道 |
|------|------|
| 可能导致崩溃、拒绝服务、越权访问、数据损坏、信息泄露 | **security(at)postgresql(dot)org**（安全团队私下处理，走协调披露流程） |
| 普通逻辑错误，无安全影响 | [官方 bug 表单](https://www.postgresql.org/account/submitbug/) 或直接发送补丁到 pgsql-hackers 邮件列表 |
| 不确定是否算安全问题 | 优先按安全问题处理，宁可谨慎；安全团队会视情况转为公开 bug |

## 第三步：撰写披露邮件/表单内容的原则

- **给出复现所需的最小信息**，不要在公开渠道（尤其是普通 bug 表单，会被公开索引）附带可以直接复制粘贴使用的完整攻击脚本
- 安全问题优先走私下渠道（security 邮箱），待官方评估、修复、协调发布后再考虑是否公开细节
- 用英文撰写发给上游社区的邮件正文（社区以英文沟通为主），可以在附件里保留中文版详细报告供内部存档
- 明确说明：
  - 复现环境（版本、配置、操作系统）
  - 复现步骤（编号列出）
  - 你认为的根因（如果有把握），但要注明这是个人判断，需要社区确认
  - 是否已经检索确认非重复报告

## 第四步：等待与跟进

- 安全问题的响应周期可能是数周到数月（协调披露流程），保持耐心，不要在响应前公开细节
- 普通 bug 表单/邮件列表通常几天到几周内会有维护者回应；如果长时间无响应，可以礼貌地在邮件列表跟进一次，附上更多细节而不是重复发送

## 绝对不要做的事

- 不要在公开渠道贴出可直接用于攻击生产环境的完整 payload/利用代码
- 不要跳过检索步骤直接提交，浪费维护者精力去处理重复报告
- 不要在报告中夸大影响面或做未经验证的安全声明
- 不要对未获授权的第三方 PostgreSQL 实例（如公有云上他人的实例）做任何动态测试
