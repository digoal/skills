# 通用问题分析统一契约（内部骨架版）

本文件定义所有"开放性/跨学科/有争议问题分析类"技能在**成文前**必须完成的
内部分析骨架。它不是最终交付物，而是最终叙事文章的骨架依据——文章可以写得
有人味、第一人称、面向小白，但每一句结论性表述都必须能倒查回本骨架中的
某一项，不能凭空生成。

**适用范围**：multi-expert-analyzer、
red-blue-debate、bug-hunter、article-fact-checker、good-ask、
industry-insight-writer、industry-deep-explainer、product-neutral-advisor、
enterprise-planner、personal-planner、state-trend-advisor 等一切以"分析开放性
问题、给出有依据的观点"为核心产出的技能。

**不适用**：纯诊断类技能、纯创作/转述类技能（book-note-writer、podcast-script 等无需本骨架）。

---

## 内部分析骨架

```markdown
## 结论 / 观点
[核心判断是什么。如果问题本身存在合理分歧，允许并列多个观点，
并标注各自成立的条件，而不是强行收敛成一个"正确答案"]

## 逻辑推演过程
[从问题拆解到结论的推导路径：问题被拆成了哪几个子问题、
各领域专家各自的推理链条是什么、如何从多个专家视角收敛/分歧到当前结论。
不是证据罗列，是推理过程本身]

## 证据链
1. [具体数据/研究/事件/专家意见] → 支持了什么判断，来自哪位专家视角
2. [具体数据/研究/事件/专家意见] → 支持了什么判断，来自哪位专家视角
...
（每条证据须标明来源，可回溯到具体链接/文献/统计口径，
不接受"大家都知道""普遍认为"这类无源表述）

## 证据权威性
| 证据 | 来源类型 | 权威性 |
|---|---|---|
| 证据1 | 官方统计/学术论文/一手数据 | 高 |
| 证据2 | 权威媒体报道/行业权威机构报告 | 中高 |
| 证据3 | 专家个人观点/行业资深人士访谈 | 中（因人而异，需看该专家在该领域的公信力） |
| 证据4 | 网络讨论/自媒体/未经核实的传闻 | 低，仅作参考不作为结论支撑 |
（跨学科问题的证据强弱差异远大于诊断类问题，此表尤其重要，
避免把一条网络传闻和一篇权威论文并列引用而不加区分）

## 结论适合的边界
[这个结论在什么范围内成立：适用的时间窗口（是否会被短期事件推翻）、
地域/文化/制度背景、问题的哪个层面（现象层/机制层/价值判断层）。
明确说明"本结论不适用于"的场景，尤其要标注哪些是事实判断、
哪些是价值判断——价值判断部分不该被包装成"客观结论"]

## 结论成立的前提条件（第一性原理拆解）
[把结论拆解到最基础的假设层，列出缺一不可的前提，包括隐含假设。
例（以"某商业模式是否可持续"为例）：
- 前提1：当前的监管环境/政策方向在可预见时间内不发生根本性变化
- 前提2：所引用的数据/统计口径在样本选取上不存在系统性偏差
- 前提3：分析建立在"理性经济人"假设上，未充分考虑非理性行为/黑天鹅]

## 前提发生变化时的新结论
[针对上面每条前提，给出"如果前提不成立，结论会如何变化"，
形成条件分支而非单一死结论。
例：
- 若前提1不成立（监管政策收紧）→ 结论应反转为"不可持续"，
  且反转的触发信号是什么（列出可观测的早期指标）
- 若前提3不成立（出现非理性挤兑/舆情事件）→ 原结论的时间窗口大幅缩短，
  风险敞口需重新评估]

## 反方 / 红队视角
[最有力的反对意见是什么，来自哪个立场/哪位专家视角，
其论证逻辑和证据是什么——不是为了反驳而反驳的稻草人，
而是真正有说服力、值得读者认真对待的对立观点。
如果红队视角未能被有效回应，应在结论中如实反映，
而不是强行压制反方以维持结论的"完整性"]

## 置信度
高/中/低，并标注：这是**共识性判断**还是**存在合理争议**。
理由：[基于证据权威性、专家视角一致程度、红队反驳的有效性综合判断——
专家间高度一致且反方证据薄弱，置信度高；专家间本身存在实质分歧，
置信度应如实标注为"中"或"存在争议"，不应为了显得笃定而虚高]

## 结论边界之外：需读者自行判断的部分
[哪些部分已经超出事实分析范畴，进入价值取向/立场选择领域——
这部分应明确交还给读者自行判断，而不是替读者做价值判断后
包装成客观结论]
```

## 生成报告前自检（技能自查用，不作为最终文章章节输出）

- [ ] 是否存在被隐藏/未被认真对待的反方观点？
- [ ] 事实判断与价值判断是否被清楚区分，价值判断部分是否被误包装成客观结论？
- [ ] 置信度是否如实反映专家间的分歧程度，而非为了显得笃定而虚高？
- [ ] 证据权威性分级是否被执行，而非把网络传闻和权威研究并列引用？
- [ ] 前提条件是否拆到了真正"缺一不可"的层次，而非表面化的免责声明？
- [ ] 最终成文（第一人称叙事版）中的每一句结论性表述，是否都能倒查回本骨架？

## 与最终成文的关系

本骨架是分析阶段（专家起草 → 事实核查 → 红队审阅）的中间产物，建议保存在
项目 `markdown/` 目录下作为附属文件（如 `xxx-analysis-skeleton.md`），
与最终对外发布的第一人称叙事文章并存。最终文章允许：

- 省略骨架中"证据权威性表格"这类结构化呈现，改为叙事中自然带出信源可信度
- 将"反方视角"融入行文而非单独成段，但不能删除反方本身
- 将"前提变化后新结论"转化为"如果……情况会不同"这类口语化表达

但不允许：脱离骨架凭空补充骨架中没有的结论、淡化骨架中标注为"存在争议"
的内容以换取更笃定的叙事效果。

## 各技能接入方式

在各技能的 SKILL.md 正文中加入引用说明，例如：

```markdown
## 分析规范

本技能在撰写最终文章前，须先按 _shared/general-analysis-skeleton.md
完成内部分析骨架（专家视角、证据分级、反方意见、前提条件等），
再基于骨架重写为面向读者的第一人称叙事文章。骨架可作为附属产物保存，
最终文章中的结论性表述均须可回溯至骨架内容。
```

建议将本文件放置于共享目录（如 `_shared/general-analysis-skeleton.md`），
与 `diagnostic-output-contract.md` 并列，作为你技能仓库中两类分析型技能
共用的两套底层契约。
