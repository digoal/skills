# 健康评分计算规则（百分制）

总分 = 缓存命中率得分(30%) + 平均执行时间得分(30%) + 全表扫描比例得分(20%) + WAL生成合理性得分(20%)

## 1. 缓存命中率得分（满分 30）

以全实例整体缓存命中率（全部 TOP SQL 的 `sum(shared_blks_hit) / sum(shared_blks_hit + shared_blks_read)`）为准：

| 命中率区间 | 得分 |
|------------|------|
| >= 99% | 30 |
| 95% ~ 99% | 24 |
| 90% ~ 95% | 18 |
| 70% ~ 90% | 10 |
| < 70% | 3 |

## 2. 平均执行时间得分（满分 30）

以全部 TOP SQL 的 `mean_exec_time` 中位数为准（避免个别极端值扭曲）：

| 中位数区间 | 得分 |
|------------|------|
| < 10ms | 30 |
| 10ms ~ 50ms | 24 |
| 50ms ~ 200ms | 16 |
| 200ms ~ 1000ms | 8 |
| >= 1000ms | 2 |

## 3. 全表扫描比例得分（满分 20）

以"单次返回行数异常 TOP"+"总扫描行数 TOP"中，经诊断确认存在全表扫描迹象的 SQL 占 TOP 10 总数的比例为准（无法通过执行计划确认时，按文本特征粗判并注明置信度）：

| 疑似全表扫描占比 | 得分 |
|------------------|------|
| 0% | 20 |
| < 10% | 16 |
| 10% ~ 30% | 10 |
| 30% ~ 60% | 5 |
| >= 60% | 1 |

## 4. WAL 生成合理性得分（满分 20）

以采集间隔内 WAL 生成总量与业务预期量级的相对水平判断（无绝对阈值，需结合实例规格）：

- 若 WAL 生成量 TOP 前 3 条 SQL 合计占全部 WAL 生成量的比例极高（如 > 80%）且为单行高频 DML，判定为"可优化空间大"，扣减该项得分至 8~12。
- 若 WAL 生成分布较均匀，或高 WAL 量的 SQL 本身就是合理的批量写入操作，得分 16~20。
- 无法判断时默认给 15 分（中性），并在报告中注明"WAL 合理性需结合业务预期进一步确认"。

## 评分区间解读

| 总分 | 画像 |
|------|------|
| 90-100 | 健康：整体负载合理，无明显紧急问题 |
| 75-89 | 良好：存在局部优化空间，非紧急 |
| 60-74 | 需关注：存在若干明显瓶颈，建议排期优化 |
| < 60 | 需紧急处理：存在严重性能风险，建议优先处理 Top 3 SQL |
