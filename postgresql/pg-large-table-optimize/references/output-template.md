## 🔍 逐表详细分析（按优化优先级降序）

每张「需要优化的大表」用如下小节结构展开：

### `<schema>.<table>` — 优先级：<高/中/低>

**基本画像**
- 总大小 / 表本体大小 / 索引总大小 / TOAST 大小
- 修正后真实大小，膨胀占比及判定依据
- 估算行数、是否分区表（及分区数/分区策略）
- 索引数量及各索引大小

**负载数据**
- 写入比率 / 更新比率 / HOT 更新效率 / DML 密度（附具体数值和计算式）
- 索引使用率 / 每次索引扫描平均行数 / 每次顺序扫描平均行数
- 索引深度：列出每个索引的估算层高，标注「偏深」的索引

**负载类型判定及依据**
- 明确写出命中了 A/B/C/D/E 中的哪一类，以及具体是哪个数值触发了判定阈值

**具体优化方案**
- 从 `references/optimization-types.md` 对应类型的模板中挑选适用条目，给出具体 SQL 模板（不代表已执行）
- 分区键/归档周期等涉及业务语义的部分，标注「需与业务方确认」

**预期收益评估**
- 定性 + 定量描述，如"分区后单表 vacuum 时间预计从 30 分钟降至 3 分钟"、"索引体积预计从 X GB 降至 Y GB"（给出估算依据，不夸大）

**实施风险及执行窗口**
- 是否需要锁表、是否需要业务停窗、建议的执行时间段（如低峰期）
- 回滚方案（如分区改造前先在测试环境验证、DETACH PARTITION 而非直接 DROP）

## ⚙️ 实例级参数调整建议

汇总所有大表分析中发现的系统级共性问题，给出统一建议（非针对单表）：

- `autovacuum` 相关全局参数（如 `autovacuum_max_workers`、`autovacuum_vacuum_cost_delay`）是否需要调整
- `work_mem` / `maintenance_work_mem` 是否偏小导致分析查询频繁溢出磁盘或索引重建缓慢
- `shared_buffers` / 操作系统 page cache 是否匹配当前热数据集大小
- 是否建议安装 `pgstattuple`（若尚未安装，可获得更精确的膨胀分析）

> 本报告中的统计数据基于 `pg_stat_user_tables` 等累积计数器（自上次统计重置以来的总体情况）。如需更精确的时段性负载分析（如仅分析业务高峰期的负载特征），建议在高峰期前后分别采集快照后做差值分析。所有涉及 DDL/维护操作的 SQL 均为建议模板，本技能未执行任何变更，请在业务方确认并安排维护窗口后手动执行。
