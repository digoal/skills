# Bug 报告模板

每一个通过人工复核、置信度为「中」或「高」的候选，都应基于本模板生成一份独立文件，
保存到 `find-bug-artifacts/reports/<简短标识>.md`。

---

## 标题

一句话概括现象与影响面。示例：`CREATE INDEX CONCURRENTLY 与并发 VACUUM 竞争下 btree 索引损坏`

## 基本信息

| 字段 | 内容 |
|------|------|
| 发现日期 | YYYY-MM-DD |
| 目标分支/版本 | 如 master @ commit abcdef1，或 REL_16_STABLE |
| 复现率 | 如 "3/10 次触发" 或 "稳定复现" |
| 置信度 | 高 / 中（说明理由，见下） |
| 类别 | 内存管理 / 并发锁 / 边界溢出 / 逻辑错误 / 资源泄漏 / 类型转换 |
| 影响 | 崩溃 / 数据损坏 / 结果错误 / 拒绝服务 / 信息泄露 / 仅代码质量问题 |

## 一句话摘要

（现象 + 触发条件 + 影响，控制在 2-3 句）

## 复现步骤

1. 构建配置：`./configure --enable-cassert ...`（列出所有非默认配置项）
2. 初始化实例：`initdb ...`，关键 `postgresql.conf` 参数
3. 准备数据/schema（最小化，只保留触发所需的部分）
4. 触发操作的精确顺序（编号列出每一步 SQL/命令，含并发时序说明）
5. 观察到的现象（错误信息、崩溃堆栈、amcheck 报错、sanitizer 输出等，附日志片段）

- 实际：...

## 根因分析（如有把握）

- 涉及源码位置：`文件路径:行号`，函数名
- 可能的根因：...（结合阶段2挖掘出的历史相似模式，说明这是否是同类问题的再次出现）
- 尚不确定的地方：如实说明哪些环节还需要社区确认

## 置信度理由

- 高：已在 sanitizer 构建下稳定复现，且能定位到具体代码路径
- 中：能复现现象，但根因分析仍有不确定性，或复现率不稳定
- （置信度为低的候选不应生成正式报告，回到 candidates.md 继续观察）

## 建议披露渠道

- [ ] security(at)postgresql(dot)org （怀疑有安全影响：崩溃/越权/数据损坏/DoS）
- [ ] 官方 bug 表单 https://www.postgresql.org/account/submitbug/ （普通功能性 bug）
- 是否已检索确认未被重复报告：是 / 否（附检索关键词和结果）

## 附件清单

- 崩溃日志 / sanitizer 输出：`find-bug-artifacts/.../xxx.log`
- 复现脚本/SQL：`find-bug-artifacts/.../repro.sql`
- 相关历史 commit（如适用）：hash + 链接
